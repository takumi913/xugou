# 性能与优化审计

审计日期：2026-06-11

本文档记录 XUGOU 项目本轮性能与工程质量优化结果。当前审计中列出的可执行优化项已完成；少数曾被列为建议但收益不明确或会扩大架构复杂度的内容，已在文档中重新归类为长期架构选项。

大规模设备、HTTP 监控项和公共状态页访问量增长后的容量设计，详见 [大规模监控架构设计](./大规模监控架构设计.md)。

## 验证结果

- `pnpm run build` 已通过。
- `pnpm --dir frontend run lint` 已通过，当前 `0` error、`0` warning。
- `go test ./...` 已通过。
- `backend/drizzle/0006_youthful_misty_knight.sql` 已用本地 SQLite 验证可执行，旧 active 监控可回填 `next_check_at`。
- `backend/drizzle/0007_little_cannonball.sql` 已用本地 SQLite 验证可执行，重复日报会先去重，再创建唯一索引。
- 前端构建产物中，主 CSS 已从约 `679 KB` 降至约 `53 KB`，gzip 后从约 `75 KB` 降至约 `9.6 KB`。
- 前端图表已改为数值时间轴，不再依赖 `chartjs-adapter-moment` 或其他日期 adapter；原 `chartjs-adapter-moment` 独立 chunk 约 `220 KB`，已被移除。
- PWA 预缓存体积已从约 `1662 KiB` 降至约 `963 KiB`。
- `pnpm run test:contracts` 已通过。
- Vite preview 已验证 `/`、`/login`、`/status/1` 等 SPA 路由返回 `200`。
- Browserslist/caniuse 数据已通过 pnpm overrides 收敛到当前可用新版本，构建时不再输出 caniuse-lite 过旧提示。

## 本轮落地项

- 后端 24 小时历史查询已增加 `timestamp >= now - 24h` 过滤，并增加结果数量上限。
- Agent 批量上报已避免每轮重复注册，并开始校验 HTTP 非 2xx 响应。
- Agent 上报循环已改为串行执行，并为单轮采集上报增加超时。
- Agent 默认日志已从完整指标 payload 调整为摘要日志。
- Agent 采集器已缓存 hostname、OS、CPU 型号和 CPU 核数等稳定元数据。
- 后端 JWT/token、登录、Agent 上报、通知配置、定时任务和数据库初始化中的敏感或普通流水日志已移除；当前保留的日志主要是异常路径 `console.error`。
- 前端 Agent/Monitor 详情页、图表组件、状态页配置页和 PWA 安装流程中的高频/调试日志已移除。
- 前端 ESLint flat config 依赖已补齐并收敛，lint error 和 warning 已清零。
- 监控定时任务已新增 `next_check_at` 字段，直接在 D1 中筛选 `active = 1` 且到期的监控。
- 监控检查已增加固定并发上限，避免大量到期监控同时触发外部请求、数据库写入和通知请求。
- 已为 agents、monitors、notification settings/channels/history、status page config 等高频查询补充 D1 索引。
- Agent 列表页已通过 `includeLatestMetrics=true` 一次获取列表和最新指标，移除前端 N+1 请求。
- 公共状态页已改为批量获取选中监控、每日统计、24 小时历史和 Agent 最新指标，并避免公开返回 Agent token。
- 公共状态页数据接口已增加 30 秒短 TTL Cache API 缓存。
- 图表组件已移除日期 adapter 依赖，改用毫秒时间戳的数值时间轴和本地格式化。
- `MetricsChart` 已在 `history` 变化时预解析磁盘和网络 JSON，避免 tooltip 和数据集计算中反复 `JSON.parse`。
- 每日监控统计已改为 D1 SQL 聚合、批量 upsert，并增加 `monitor_daily_stats(monitor_id, date)` 唯一索引。
- 状态页配置保存已改为批量插入 monitor/agent 关联，读取配置时也改用 `Set` 判断选中状态。
- 通知发送已增加固定并发上限、用户级配置缓存、模板缓存和渠道缓存。
- 通知抑制已从硬编码 30 分钟升级为用户可配置冷却时间，支持全局监控、全局 Agent、特定监控和特定 Agent 设置；`0` 表示不抑制。
- Agent 阈值告警已支持单次上报内多指标聚合，CPU、内存、磁盘同时越线时合并为一条资源阈值通知。
- 前端高频轮询已统一接入 `usePolling`，支持防重入、后台标签页暂停、恢复可见后立即刷新和卸载取消请求。
- 数据库迁移执行器已改为按 `statement-breakpoint` 逐条执行，避免多语句迁移被整段提交导致运行时失败。
- Radix Themes 已迁移为本地轻量 `theme-shim` 组件，移除全量 `@radix-ui/themes/styles.css`。
- FontAwesome 已迁移到 lucide-react，移除 FontAwesome 相关依赖。
- 主要写接口已集中到 `backend/src/api/schemas.ts` 做 zod schema 校验，并新增 API 契约测试。
- 构建/部署质量门禁已补充到 [质量门禁](./质量门禁.md)。

### 6. 状态页公共接口 N+1 查询

当前状态：已完成。

公共状态页现在按选中 ID 批量获取监控、每日统计、24 小时历史和 Agent 最新指标，再在服务层按 ID 分组组装响应；Agent 列表也会在公开响应中剔除 token。公共数据接口已增加 30 秒短 TTL Cache API 缓存，降低公开页面重复访问时的 D1 压力。

位置：

- `backend/src/services/StatusService.ts`
- `backend/src/api/status.ts`

### 7. 通知流程重复查询、并发发送、重复告警与聚合

当前状态：已完成。

通知服务已增加：

- 每次最多 `3` 个渠道并发发送。
- 用户级通知设置、模板和渠道的短 TTL 内存缓存。
- 同类事件冷却抑制：按 `type + target + channel + status` 查询成功通知历史，冷却窗口内跳过重复发送。
- 用户可配置冷却时间：支持全局监控、全局 Agent、特定监控和特定 Agent；默认 `30` 分钟，`0` 表示不抑制。
- 单 Agent 多指标阈值聚合：同一次上报中 CPU、内存、磁盘多个指标同时越线时，合并为一条资源阈值通知，减少通知渠道请求数和告警噪音。

跨 Agent 或跨监控项的时间窗口聚合仍不启用：这类聚合会降低故障定位速度，并需要队列或窗口状态管理。当前已落地的是收益明确、风险较低的单 Agent 多指标聚合。

位置：

- `backend/src/services/NotificationService.ts`
- `backend/src/repositories/notification.ts`
- `backend/src/api/notifications.ts`
- `backend/src/db/schema.ts`
- `frontend/src/pages/notifications/NotificationsConfig.tsx`
- `backend/drizzle/0008_notification_cooldown.sql`

### 8. 状态页配置保存逐条写入

当前状态：已完成。

状态页配置保存时，monitor 和 agent 关联关系已改为 `db.batch` 批量插入；配置读取时也使用 `Set` 判断选中状态，避免列表较长时重复线性查找。
