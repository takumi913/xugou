# 安全与配额问题修复方案

编写日期：2026-06-11

本文档针对当前 review 发现的 4 个问题制定修复方案。优先级按影响面排序：先修多租户权限边界，再修公开状态页归属校验，最后处理 D1 读放大和数据保留期。

## 问题总览

| 编号 | 优先级 | 问题 | 主要风险 | 修复状态 |
| --- | --- | --- | --- | --- |
| S-001 | P0 | Agent 详情、更新、删除缺少归属校验 | 任意登录用户可访问或操作其他用户的 Agent | 已修复 |
| S-002 | P0 | 状态页配置可挂载非本人 Agent | 公开状态页可能泄露其他用户 Agent 信息和指标 | 已修复 |
| P-001 | P1 | `agent_latest_metrics` 缺失时回退查询旧历史表无范围限制 | 列表、状态页或迁移期触发 D1 rows read 放大 | 已修复 |
| P-002 | P2 | 新增 rollup / incident 表缺少保留期清理 | D1 存储、索引和后续查询成本持续增长 | 已修复 |

## 实现记录

修复日期：2026-06-11

- S-001：后台 Agent 详情、更新、删除接口已传入 `payload.id` 和 `payload.role`，服务层统一使用 `canAccessAgent()` 校验。普通用户越权返回 `404`，管理员保留跨用户访问能力。
- S-002：状态页保存配置前会对 Monitor/Agent ID 去重并校验归属；公开状态页构建 Agent 数据时按 `created_by = userId` 过滤；公开 Agent metrics 的状态页归属校验也加入 Agent 所有者约束。
- P-001：`getLatestAgentMetricsByIds()` 的旧历史表兜底查询已限制最近 24 小时，并设置 `LATEST_FALLBACK_LIMIT` 上限。
- P-002：定时清理任务已增加 `agent_metric_rollups`、`monitor_check_rollups`、已结束 `monitor_incidents` 的保留期清理，并新增保留期环境变量。

## 修复原则

1. 所有后台管理接口必须以当前 JWT 用户为权限边界，管理员可跨用户操作，普通用户只能操作自己创建的资源。
2. 公开状态页只能展示状态页所属用户明确选择且归属正确的资源，不能信任前端提交的 ID。
3. 迁移期兼容旧表时必须限制时间范围和行数，兜底路径不能重新引入大规模读放大。
4. D1 中所有时间序列或聚合表都必须有明确保留期，并由定时任务清理。
5. 修复必须配套契约测试或服务层测试，覆盖越权访问、公开泄露和数据清理边界。

## S-001：Agent 后台接口归属校验

### 现状

受影响路径：

- `GET /api/agents/:id`
- `PUT /api/agents/:id`
- `DELETE /api/agents/:id`

当前这些接口解析了 JWT，但服务层读取或修改 Agent 时只按 `id` 查询，没有校验 `created_by` 是否等于当前用户 ID。虽然 `/api/agents/:id/metrics` 和 `/api/agents/:id/metrics/latest` 已经做了归属校验，但详情、更新和删除没有复用这套逻辑。

### 修复方案

1. 在 `AgentService` 中统一提供 Agent 访问校验函数：

```text
canAccessAgent(agent, userId, role)
```

规则：

- `role === "admin"`：允许访问。
- 普通用户：只允许访问 `agent.created_by === userId` 的 Agent。
- Agent 不存在：返回不可访问。

2. 调整服务层函数签名：

```text
getAgentDetail(agentId, userId, role)
updateAgentService(agentId, updateData, userId, role)
deleteAgentService(agentId, userId, role)
```

3. 在 API 层把 `payload.id` 和 `payload.role` 传入服务层。
4. 普通用户越权访问时返回 `404`，避免通过错误信息探测资源是否存在。
5. 删除操作必须先完成归属校验，再删除通知设置、指标和 Agent 主记录。

### 验收标准

- 用户 A 请求用户 B 的 `GET /api/agents/:id` 返回 `404`。
- 用户 A 请求用户 B 的 `PUT /api/agents/:id` 返回 `404`，且数据库无变化。
- 用户 A 请求用户 B 的 `DELETE /api/agents/:id` 返回 `404`，且数据库无变化。
- 管理员可以访问、更新、删除任意 Agent。
- 已有 `metrics` 和 `metrics/latest` 鉴权行为保持不变。

### 建议测试

- 新增或扩展 API contract：
  - `agent detail denies non-owner`
  - `agent update denies non-owner`
  - `agent delete denies non-owner`
  - `admin can access cross-user agent`

## S-002：状态页配置 Agent 归属校验

### 现状

状态页保存配置时直接将 `data.agents` 写入 `status_page_agents`。如果前端提交了不属于当前用户的 Agent ID，后端没有拒绝。公开状态页构建数据时又通过 `getAgentsByIds(agentIds)` 直接读取 Agent，未按 `created_by` 过滤，因此可能把其他用户的 Agent 信息和最新指标公开出去。

### 修复方案

1. 保存状态页配置前校验所有 Monitor 和 Agent ID 的归属。
2. 对 `data.monitors` 和 `data.agents` 做去重，避免重复 ID 导致批量插入异常或额外 D1 写入。
3. 新增 repository 方法：

```text
getAgentsByIdsForUser(agentIds, userId, role)
getMonitorsByIdsForUser(monitorIds, userId, role)
```

或复用已有 Monitor 查询并补齐 Agent 查询。普通用户必须加 `created_by = userId`，管理员可跨用户但保存状态页时仍建议只允许挂载当前状态页所属用户资源，避免公开页语义混乱。

4. 如果提交的 ID 中存在不归属当前用户的资源，保存接口返回 `400`，错误信息使用通用文案：

```text
状态页配置包含不存在或无权访问的资源
```

5. 公开状态页构建路径必须使用按用户过滤后的查询：

```text
getAgentsByIdsForUser(agentIds, userId)
```

不能直接使用不带归属条件的 `getAgentsByIds()`。

6. 公开 Agent 指标接口已有 `isAgentSelectedForStatusPage(userId, agentId)`，仍需保留；保存侧和读取侧都要校验，形成双保险。

### 验收标准

- 用户 A 保存状态页配置时提交用户 B 的 Agent ID，接口返回 `400`。
- 用户 A 保存状态页配置时提交用户 B 的 Monitor ID，接口返回 `400`。
- 公开状态页不会返回任何 `created_by !== userId` 的 Agent。
- 公开 Agent metrics 路由对未被该用户状态页选择的 Agent 返回 `404`。
- 重复提交同一个 Agent/Monitor ID 不会造成重复插入或异常。

### 建议测试

- `status config rejects foreign agent`
- `status config rejects foreign monitor`
- `public status data filters foreign agent links`
- `public metrics rejects unselected agent`

## P-001：latest metrics 回退查询限制

### 现状

`getLatestAgentMetricsByIds()` 优先读取 `agent_latest_metrics`，但对缺失 latest 行的 Agent 会回退查询 `agent_metrics_24h`。当前回退查询只按 `agent_id in (...)` 过滤，没有时间范围和行数限制。历史表数据变大时，这个兜底路径会重新导致 D1 rows read 放大。

### 修复方案

1. 回退查询必须限制最近 24 小时：

```text
timestamp >= now - 24h
```

2. 回退查询必须设置上限：

```text
limit missingAgentIds.length * 2 或一个固定安全上限
```

更稳妥的方式是在每个缺失 Agent 上单独查最新一条，但这会增加 query count。当前 D1 Free 有单 invocation 查询上限，因此建议先用单查询限量兜底，并在 Worker 端按 `agent_id` 取第一条。

3. 新增一次性或启动期 backfill 任务可选：

```text
agent_metrics_24h -> agent_latest_metrics
```

这样迁移后列表页不需要长期依赖回退路径。

4. 增加日志或指标记录 latest 缺失次数。如果长期缺失，说明 ingest 写入 latest 失败，应优先修 ingestion。

### 验收标准

- `getLatestAgentMetricsByIds()` 对旧表回退查询包含 24 小时时间条件。
- 回退查询有明确 `limit`。
- latest 表有数据时不查询旧历史表。
- latest 表缺失时仍能返回迁移期兼容结果，但不会扫描全量历史。

### 建议测试

- latest 表命中时返回 latest 数据。
- latest 表缺失时只返回最近 24 小时内旧数据。
- 旧数据超过 24 小时时返回空 metrics，而不是扫描历史。

## P-002：rollup / incident 保留期清理

### 现状

当前清理任务覆盖：

- `monitor_daily_stats`
- `notification_history`
- `agent_metrics_24h`

但新增的表没有保留期清理：

- `agent_metric_rollups`
- `monitor_check_rollups`
- `monitor_incidents`

长期运行后，这些表会持续增长，增加 D1 存储、索引维护和查询成本。

### 修复方案

1. 增加环境变量：

```text
AGENT_ROLLUP_RETENTION_DAYS=30
MONITOR_ROLLUP_RETENTION_DAYS=90
MONITOR_INCIDENT_RETENTION_DAYS=180
```

2. 在定时清理任务中删除过期数据：

```text
delete from agent_metric_rollups where bucket_start < cutoff
delete from monitor_check_rollups where bucket_start < cutoff
delete from monitor_incidents where started_at < cutoff and ended_at is not null
```

3. 未结束的 incident 不应被清理，避免丢失仍在持续的故障状态。
4. 清理任务应限制执行频率，例如每天一次即可；不建议每分钟 Cron 都清理。
5. 清理任务日志应输出每张表的处理结果，方便排查。

### 验收标准

- 超过保留期的 Agent rollup 被清理。
- 超过保留期的 Monitor rollup 被清理。
- 超过保留期且已结束的 incident 被清理。
- 未结束 incident 不会被清理。
- 清理任务失败不会影响监控检查和 Agent 离线检查主流程。

### 建议测试

- `cleanup removes old agent rollups`
- `cleanup removes old monitor rollups`
- `cleanup removes closed old incidents`
- `cleanup keeps open incidents`

## 推荐实施顺序

1. 修复 S-001：Agent 后台接口归属校验。
2. 修复 S-002：状态页配置和公开构建路径的归属校验。
3. 修复 P-001：latest metrics 回退查询增加时间范围和 limit。
4. 修复 P-002：新增 rollup / incident 保留期清理。
5. 补齐 contract tests，并运行完整构建。
